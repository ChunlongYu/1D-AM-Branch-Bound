# new_oracle —— 全新单机 oracle(Type I/II 分支 + 合法下界 + 支配 + 自适应 BFS&DFS)

单机一维(面积约束)AM 批调度,目标 $\min\sum_j T_j$。**自包含**实现(不依赖 `src/`、`references/`),按 `docs/Dominance_Rules_TypeI_II_Oracle.md` 落地。

## 设计要点

- **分支**:Type I(封口当前批次、开新批次)/ Type II(并入当前开口批次,下标递增 + 面积可行)。
- **合法下界**(修正了原 `LB_par` 的错误):每个未排零件的完工下界取"并入开口批次"与"另开新批次"的较小完工增量;开口批次零件按当前完工 $C_{now}$ 计;已封口延误精确。
- **支配 类1/2**:键 =(已排集合, **开口批次内容**);每个键维护 $(t^-,\,TT^{\mathrm{cl}})$ 的二维帕累托前沿(比较已封口延误,非总延误)。
- **支配 类3a**:同父同零件"合并 vs 封口",当开口批次零件合并后都不延误且面积不紧时,合并支配封口,丢弃封口子节点。
- **自适应 BFS&DFS**:两池共用一个开放列表——`pq`(best-first,收紧界)与 `stk`(DFS 深潜,迅速到叶子拿上界、控内存)。总规模 `>N_MAX` 切 DFS、`<N_MIN` 切回 BFS(滞回);DFS 每次深潜从 `pq` 最优节点播种,故 `pq` 持续被抽干。节点顺序不影响正确性。

## 构建 / 运行

```bash
g++ -std=c++17 -O2 -o oracle oracle.cpp
./oracle ../data/15part.txt 60            # 实例  时限(秒)  [N_MAX] [N_MIN]
```

## 关于"宽分支冗余 / Type I 首件择优"的结论(重要)

问:能否给 Type I 的"开批次首件"加跨批次对称/择优规则,减少冗余分支?

实测与分析结论:

1. **批次内的顺序冗余早已消除**:Type II 的 `j > oMax` 规则保证每个批次只按下标递增构建一次(`{1,2}` 只由"先1后2"生成,"先2后1"被直接挡掉)。
2. **`[{1},{2}]` vs `[{2},{1}]` 不是重复**,是两个不同的解(批次串行,完工时间不同);且**每个有序调度只有唯一生成路径**(批次首件被强制为该批次最小下标),所以宽分支**没有重复枚举同一调度**的冗余。
3. **不同批次"排序"造成的冗余,已被现有支配前沿吃掉**:凡是"前缀不同、开口批次相同"的两个节点(绝大多数排序差异)都落在同一个键 `(已排集合, 开口批次)` 上,按 `TT^cl` 取优——更差的排序直接被剪。
4. 我额外实现并实测了一条**可证合法的相邻批次交换支配**(封口当前批次若不如与前一批次交换,则丢弃其 Type-I 子节点):
   - 正确性:`data/` 全部算例目标值仍与 `src/` 完全一致;
   - 但**展开节点数(popped)几乎不变**(11–15part:0~0.3%),因为这些子节点本来就会被下界/前沿剪掉;
   - 它的作用是**更早**剪掉这些注定要死的子节点(15part:生成子节点 8.6M→6.0M,interch_prune≈2.6M),从而省下它们的下界计算与前沿查表,**wall-time 约快 10%**(15part 1.32s→1.19s,14part 0.83s→0.75s)。

一句话:**真正的"对称/排序冗余"在当前 oracle 里已经被 `j>oMax` + 支配前沿消除了**;再加首件择优/交换规则不会缩小搜索树,只能作为"提前短路注定被剪的子节点"的小幅常数级加速(~10%)。默认开启(第 5 个参数 `1`,设 `0` 可关闭对比)。

## 位置下界 $LB^{pos}$(已接入,默认开 + 浅层门控)

按 `docs/Incremental_Branching_Bounds.md` §4.1 的 "iron" 形式接入(面积→批次数 $\beta_k$ + 体积 + 划分最小高度项),节点界改为
`LB = 已封口(精确) + 开口批次 floor + max(LB_par, LB_pos)`。$LB^{pos}$ 捕捉了 $LB^{par}$ 缺的"剩余件必须共用单机、排在后面的完工很晚"这层耦合。

- **正确性**:`data/` 全部算例目标值仍与 `src/` 完全一致(含 10part_2/10part_3)。
- **节点显著下降**,且随 n 增大收益变大;但**逐节点算 $LB^{pos}$ 较贵**(4 次排序 + O(q)),小 n 上反而更慢——存在 n≈15–16 的交叉点。
- **浅层门控**(只在 `|R| ≥ QFRAC·n` 的浅节点算 $LB^{pos}$,默认 `QFRAC=0.5`):浅节点少但收紧最多,深节点(海量)直接跳过,**两头通吃**——既保住节点削减,又消掉小 n 的开销。

| 实例 | pos 关 (pop / t) | pos 全开 (pop / t) | **浅层门控 0.5 (pop / t)** |
|---|---|---|---|
| 14part | 534k / 0.67s | 381k / 0.95s | 436k / **0.67s** |
| 15part | 873k / 1.24s | 466k / 1.33s | 731k / **1.09s** |
| 16part(合成) | 10.2M / 15.1s | 5.2M / 14.0s | 7.3M / **11.6s** |
| 17part(合成) | 12.8M / 19.0s | — | 7.6M / **14.1s** |

净效果:浅层门控版在 n≥15 上**更快**(15part −12%、16part −23%、17part −26%,且随 n 继续放大),小 n 与关闭时持平。命令行第 6/7/8 参数分别为 `pos(0/1)`、`gamma`、`qfrac`,可做 A/B 与调参。

> 备注:`docs/Incremental_Branching_Bounds.md` §8 当年"不要替换 submask oracle"的结论是针对**分支方案选择**(Python 原型按 bound-eval 计),与"在本增量 oracle 里加 $LB^{pos}$"无关——后者在当前 C++ 实现里是净增益。
