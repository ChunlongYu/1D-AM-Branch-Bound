# Zixuan 新分支单机 Oracle 评测（vs 原 ORACLE）

日期：2026-06-18 · 范围：`references/Zixuan_codes/` 单机批调度 B&B（Type I/II 增量分支 + 串行 DP 界 + 支配规则）对比 `src/`（原 oracle Φ）。

## 结论（先说要点）

1. **能跑**：代码可编译、可运行（仅一个无害的 `#endif` 警告）。
2. **小规模上并不优于原 ORACLE**：在目标值一致的算例上，新分支探索的节点更多、耗时更长（1.3–2×）。
3. **更严重的是有正确性 bug**：在部分小算例上返回**次优（偏大）**的总延误，即剪掉了最优解。
4. 根因已定位：**下界函数 `compute_unassigned_lower_bound` 对自己的 Type I/II 分支不是合法下界**——它给每个未排零件都计了一次完整 setup `ST` 和完整高度项 `UT·h`，但 Type II 允许零件并入“当前未封口批次”，边际代价远小于此。于是下界**高估**，触发 `LB ≥ UB` 误剪，丢掉最优。
5. 分支**设计本身没问题**：用 Python 精确复刻 Type I/II 枚举规则可得到真最优（见验证），所以这是实现层（界）的 bug，不是算法思想的问题。

## 正确性验证（为何确定 81.7384 才是最优）

- 原 ORACLE 给出的最优排程（10part_2）：B0={0,5,6}, B1={1,3,4,8,9}, B2={2,7}。**独立用 Python 按相同公式重算**,面积可行,总延误 = 81.7384(与原 oracle 一致)。
- **独立随机搜索**（40 万次）在 10part_2 上的最好值也是 81.7384,从未低于它,且大量解低于新分支的 98.5656。
- **精确复刻 Type I/II 分支规则**（Python,带合法的“累计延误单调”剪枝）得到树内真最优 = 81.7384——证明分支树**包含**最优,是搜索把它剪没了。

## 根因验证

- 关闭新分支的全部剪枝做纯枚举：30s 内最好值降到 87.87 并仍在下降（<98.57）——说明被剪掉的分支里确有更优解 ⇒ 剪枝在误剪。
- 把下界改成合法形式（每个未排零件只计 `VT·v`,不再强加 `ST` 与 `UT·h`)后,新分支在 10part_2、10part_3 上**恢复正确**(81.7384 / 136.74),但节点暴涨(196k / 131k),耗时 1.0–1.3s——印证“原界因高估而误剪;换成显然合法的界则正确但很松、很慢”。

## 建议

- 该“新分支”当前**不能用于实验**：在小规模上既慢又会给错最优。
- 修复方向是**重做下界使其对 Type II 合法**：未排零件的边际完工增量不应固定收一次 `ST + UT·h`,应考虑“可并入当前开口批次”的更低代价（或退用 `src/` 中已验证合法的 `compute_total_lower_bound` 思路）。修好后再比节点/时间才有意义。
- 注意代码里 `compute_unassigned_lower_bound2`（串行 DP 界）、支配规则当前均被注释关闭;真正生效的就是上面这个并行界。若日后启用 DP 界,需同样先证其为合法下界。

## 与论文 NodeGenerationNew 的对照（下界部分）

论文里写了三种界:并行界 $LB_{par}$、串行界 $LB_{ser}$、混合界 $LB_{hyb}$,外加支配规则。逐条对照代码:

### 1. 并行界 $LB_{par}$ —— 代码与论文一致,但**论文公式本身是错的**

论文(III-E-1):
$$LB_{par}=TT_{partial}+\sum_{j\in\mathcal{R}}\max\{0,\;C_{now}+S+Vv_j+Uh_j-d_j\}$$
代码 `compute_unassigned_lower_bound`:`c = completion_time + (ST + VT*v_j + UT*h_j); tard += max(0, c - D_j)`。
**完全一致**——代码忠实实现了论文公式。

但这个界**不是合法下界**。论文给的理由(“parts 并行 vs 串行,所以乐观”)只考虑了“并行/串行”这一面,**漏了自己定义的 Type II 分支**:剩余零件可以并入“当前未封口的批次”,此时它**不付新的 setup $S$**、且高度项只增 $U\max(0,h_j-H_b)$。而公式给每个剩余零件都按“在 $C_{now}$ 处新开一个批次、付满 $S$ 和 $Uh_j$”计完工时间,于是**高估**了能并入开口批次的那些零件的完工时间,导致 $LB_{par}$ 可能**大于子树真最优** ⇒ 误剪最优解。

**最小反例**(机器 $S=2,V=0.03,U=0.7$):
零件 A($v{=}10,h{=}30,d{=}1000$)、B($v{=}10,h{=}2,d{=}25$)。节点 N 的开口批次为 {A},$C_{now}=23.3$,$TT_{partial}=0$,$\mathcal{R}=\{B\}$。
- 论文/代码:$LB_{par}(N)=\max(0,\,23.3+2+0.3+1.4-25)=2.0$
- 真子树最优 = 0(Type II 合成批次 {A,B},完工 $=2+V\cdot20+U\cdot30=23.6\le25$,B 不延误)
即 $LB_{par}(N)=2.0>0=$ 真最优。当 $UB$ 略大于 0 时,这个含零延误解的节点会被错误剪掉。

这正是 10part_2 / 10part_3 返回次优的原因(已在前一节用独立枚举/随机搜索/改界后恢复正确三重验证)。

### 2. 串行界 $LB_{ser}$ —— 论文证明同样有漏洞(代码中该函数 `compute_unassigned_lower_bound2` 当前被注释关闭)

论文把 $S+Uh_{\min}$ 对整组剩余零件只收一次,体积项沿序列累加,然后解 $1\Vert\sum T_j$。证明的关键句“$C_j^{real}$ 至少含一个 setup 项和一个高度项”对**并入开口批次的零件不成立**——该零件的 setup 已含在 $C_{now}$ 里、不再单独计。整个推导默认“剩余零件都在 $C_{now}$ 之后另开新批次”,**仍然忽略 Type II 合并**。所以 $LB_{ser}$ 在“开口批次非空”的节点上同样不是合法下界。(代码里这条及 $LB_{hyb}$ 的 delta 触发逻辑、支配规则目前都被注释掉,实际只跑 $LB_{par}$。)

### 3. 支配规则 —— 同源缺陷

论文以 $(\mathcal{S}_u,C_u,TT_u)$ 为键,称“$\mathcal{S}_u=\mathcal{S}_v$ ⇒ 两节点后续连续相同、追加批次加工时间相同”。但有了 Type II,**已排集合相同、但开口批次构成不同**的两个节点,其 Type II 后续并不相同(开口批次的体积/高度/最大下标不同),所以“追加加工时间相同”不成立,支配也不严格。(代码中支配同样已注释。)

### 小结(论文 vs 代码)

- 代码 = 论文公式:**一致**($LB_{par}$ 逐项对应;$LB_{ser}$/$LB_{hyb}$/支配在代码里已注释,仅 $LB_{par}$ 生效)。
- 是否有错:**有**。$LB_{par}$ 与 $LB_{ser}$ 的“乐观/松弛”论证都**漏掉了 Type II(并入开口批次)这一更便宜的延拓**,因此都不是合法下界;支配规则的证明也基于同一错误假设。后果是**会剪掉最优解、给出偏大的 TT**(silent bug:论文 Table II 用的是另一套 TF/RDD 生成算例,恰好没触发,所以看起来都 “Opt.” 且与 Gurobi 一致;但 repo 的 10part_2/10part_3 直接触发)。
- 修法:界里对“可并入当前开口批次”的零件不得强加新 setup/满高度。可行的合法替代是按“该零件并入开口批次”和“另开新批次”两种延拓取**更小的完工增量**作为其完工时间下界;或退回 `src/` 中已验证合法的 `compute_total_lower_bound`。
